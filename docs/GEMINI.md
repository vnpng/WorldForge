# Gemini 项目指令集 (WorldForge)

## 1. 核心行为准则
- **直接修改**：在常规功能开发时，请直接修改并覆盖原文件，无需询问确认；若是排查报错，则严格遵守下方的“遇到错误”条款。
- **强制备份**：在对任何现有文件执行修改前，必须先创建一个备份，命名为 `原文件名.bak`。若备份已存在则直接覆盖。
- **严禁偷懒省略**：修改现有文件时，必须保证写入的是包含原逻辑的完整代码。绝对禁止使用 `// ...此处省略...` 或类似占位符，防止原有功能被破坏或覆盖丢失。
- **详细注释**：为所有生成的代码添加详细的中文注释，解释每一段逻辑的目的和实现方式。
- **阶段锁定**：严格按架构说明书（Tech Spec）的阶段顺序执行，每完成一个阶段必须停下等待确认，不得自行推进到下一阶段。禁止一次性输出所有阶段代码。
- **回复语言**：始终用中文回复，保持简洁，不废话。
- **遇到错误**：遇到报错时，先分析原因，再提出修复方案，不要直接动手修改，等待确认。

## 2. 架构与代码规范 (极度重要)
- **目录守则**：严格遵守前后端分离目录。前端文件仅限存放在 `static/`，后端逻辑仅限存放于根目录，持久化数据仅限 `data/`。
- **UI 视觉隔离**：修改前端代码时，绝对禁止破坏现有的双核视觉隔离体系（RPG 模式为深黑终端风全圆角，Chat 模式为现代版灰蓝风全圆角）。使用 Tailwind 布局时，务必保证同一行组件的高度、大小、间距严格对齐。

## 3. 文档与执行偏好
- **文档同步**：当项目每个大版本更新后，必须主动在 `docs/WorldForge_全局架构白皮书.md` 中追加更新日志（Changelog）和技术变更结构。仅限追加模式，绝对禁止删除或覆盖旧版本的历史记录。
- **命令优先**：若涉及系统状态查询，优先使用 `/run` 等系统命令获取实时数据。
